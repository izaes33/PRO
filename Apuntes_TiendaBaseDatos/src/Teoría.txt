A la hora de trabajar contra base de datos, lo primero que tenemos que tener en cuenta es si vamos a trabajar con
BBDD relacionales (Sql) o no relacionales (NoSql)

Base de datos
	NO-SQL 	-> mongo -> colecciones -> documentos -> JSON -> no hay relaciones directas
	SQL -> mysql/mariadb - postgresql - sqlite - oracle	- tablas - registros (PK) -> relaciones (FK)


Java se comunica con el servidor de la base de datos
Java   --------------------------------->   Servidor de base de datos
	   ---------SRVWeb+ORM-------------->

FORMAS DE REALIZAR LA CONEXIÓN:
(La tercera lo hace todo "por detrás", pero para entender la 3ª hay que entender la 2ª,
y para entender la 2ª hay que entender la 1ª)


	1 - forma directa: JDBC -> Driver, ejecuta sentencias SQL sobre una conexion a base de datos
		ejecucion "plana" de querys

		4 objetos implicados:
		- Connection:           -> representa el enlace físico con la base de datos -> uri: jdbc:mysql://localhost/miDB
		(jdbc es la primera primera parte de la URI (protocolo de enrutamiento), al igual que otros protocolos pueden ser:
		ftp:// smb:// nfs:// ssh:// file: (en protocolos de internet)). Lo siguiente que compone la URI es el tipo de
		DBS que vamos a utilizar (en nuestro caso mysql), la URL del servidor (127.0.0.1 o localhost; o
		el DNS que tengamos si es una conexión externa) y el puerto (3306 por defecto).

		- Statement:            -> permite ejecutar consultas SQL "planas". Es vulnerable a SQL Injection.

		- PrepareStatement:     -> Ejecuta consultas parametrizadas (usando ?). Previene la inyección de código malicioso
		al gestionar los datos en el lado del cliente.

		- ResultSet -> Almacena los registros obtenidos tras una sentencia SELECT.


	2 - ORM -> Hibernate (ORM Java) -> Nivel intermedio donde se mapean objetos directamente a tablas sin escribir SQL
	manualmente


	3 - modelo de 3 capas -> SpringBoot -> Nivel avanzado que independiza totalmente el cliente, el servidor y la base de datos.

“XAMPP en Docker” es una fase intermedia, sirve para Aprender Docker sin complicarte y mantener un entorno familiar.
Pero en proyectos reales se evoluciona a:
Nginx en vez de Apache - Bases de datos separadas - Microservicios

Cuando trabajamos con el entorno XAMPP replicado en Docker, nos conectamos a xamp_db, que es la base de datos a la que
se accede desde DBeaver.

XAMPP-> Interfaz web que se basa en APACHE; conecta a sistema gestor de base de datos: phpMyAdmin, FTP o servidor web
DOCKER-> Levanta programas BBDD en local independizando servicios para hacer el desarrollo lo más real posible

Diferencia entre levantar en XAMPP vs Docker:
•  XAMPP: corre nativo en Windows (más simple/rápido para empezar). - Docker: corre aislado en contenedores
(más reproducible y portable entre máquinas).
•  Versionado: en Docker fijas versiones exactas (mysql:8.0, php:8.2-apache); en XAMPP dependes del paquete instalado.
◦  XAMPP usa su propia data local. - Docker usa el volumen mysql8_data.
•  Puertos: pueden convivir si usan puertos distintos (como ahora).

DBEAVER-> SGBD que permite levantar una base de datos de cualquier tipo con un servidor propio
utilizando un protocolo de conexión (en nuestro caso: jdbc).
Es una herramienta gráfica para trabajar con bases de datos que permite ver, editar y consultar bases de datos
sin usar comandos.
Sirve para conectarte a motores como: MySQL, PostgreSQL, SQLite, Oracle, etc.

Levantas MySQL en Docker: docker run -p 3306:3306 mysql
Abres DBeaver, te conectas con: Host: localhost - Puerto: 3306
DBeaver se conecta al MySQL o al motor de BD que está dentro del contenedor

🔹 1. XAMPP: entorno local clásico (todo en tu máquina)
Con XAMPP instalas directamente en tu PC: Apache (servidor web), MySQL / MariaDB (base de datos), PHP
Funciona en tu máquina… pero no garantiza que funcione igual en producción; si otro dev tiene otra versión de
MySQL/PHP → problemas. Si subes a servidor Linux y tú usas Windows → más problemas.

🔹 2. Docker: entorno aislado en contenedores
Con Docker no instalas servicios directamente, sino que levantas contenedores (mini entornos independientes).
localhost desde tu PC ≠ localhost dentro de Docker - Cada contenedor es “su propio mundo”

Ejemplo:
Un contenedor → MySQL
Otro contenedor → PHP
Otro → Nginx

Docker es “más real” porque imita cómo funcionan los sistemas en producción hoy en día:
En producción moderna hay contenedores o microservicios (Servicios separados DB, backend, frontend)

Docker sustituye el uso de máquinas virtuales para desarrollo, porque ofrece aislamiento de forma ligera y rápida.
Antes usabas VM para: Tener Linux en Windows, islar entornos y evitar conflictos. Ahora Docker hace eso mismo:
Aísla dependencias, replica entornos reales y funciona igual en todos lados sin cargar con otro sistema operativo entero
En algunos casos (sobre todo en Windows/Mac), Docker sí usa una mini VM interna, pero tú no la gestionas

En XAMPP se tienen varias piezas:
Apache HTTP Server → sirve páginas web (HTTP)
MySQL → gestiona bases de datos
PHP → ejecuta lógica del servidor

**MySQL NO es una web, es un servidor-motor de base de datos,
funciona por TCP/IP (puerto 3306), no por HTTP, no necesitas navegador para usarlo
Puedes conectarte con: línea de comandos, apps como DBeaver, scripts en PHP, Python, etc.

Apache es un servidor web que recibe una petición web y ejecuta PHP.

PHP se conecta a MySQL y devuelve el resultado al navegador.

🔄 Flujo real (importante)
Navegador → Apache → PHP → MySQL
                         ↓
                  devuelve datos

(Puedes apagar Apache y MySQL sigue funcionando, ypuedes usar Apache sin MySQL (páginas estáticas)).

Apache ≠ Internet; Apache es un servidor web, pero:
Puede servir páginas en local (localhost) sin necesidad de conectarse a internet
(No estás “conectándote a una web externa”). Estás accediendo a tu propio servidor en tu PC

Aunque aparezca “xampp” en Docker, sigue siendo Docker puro:

xampp_db → base de datos; usa MySQL
xampp_phpmyadmin → Interfaz web para la BD; usa phpMyAdmin
xampp_web → Servidor web; probablemente Apache + PHP

Diferencia clave
XAMPP real -> Instalado en tu sistema -> Todo junto (Apache + MySQL + PHP), puertos por defecto
“XAMPP en Docker” -> Separado en contenedores, cada servicio independiente, puertos personalizados
En vez de: “instalo XAMPP”, se hace: “levanto Apache, MySQL y phpMyAdmin por separado”
(Es una imitación modular de XAMPP, no el original).

Por qué se hace esto -> Porque en Docker: 1 contenedor = 1 servicio = Más control, más parecido a producción

Resumen: No se está usando XAMPP dentro de Docker, se están usando contenedores que hacen lo mismo
(El nombre “xampp” es solo etiqueta, no el software real).
